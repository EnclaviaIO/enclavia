# synchronizer

In-enclave storage synchronization. Replicates the LUKS-encrypted volume's freshness state across a mutually-attested vsock mesh between enclaves, so storage-enabled enclaves can quiesce, fail over, and recover without losing data or accepting stale reads.

The mesh is **mutually attested**: each peer connects to the others over vsock via the host-side `mesh-host` relay daemon (shipped from enclavia-crates, out of this repo's scope; wire format in `enclavia-protocol::mesh`), wraps the byte stream in end-to-end Noise, and only accepts the peer's PCRs after verifying its Nitro attestation. From the storage layer's point of view, the synchronizer is a small Raft cluster that tracks the latest transaction id per superblock; from the security layer's point of view, it is a closed group of identical EIFs talking to themselves.

Current scope (incremental):
- Single-node listener binary and protocol skeleton (shipped).
- Transition authorized by a #47 upgrade chain link. The NEW enclave submits it: at cutover the old enclave has stopped, so the new enclave boots, attests as `new_key`, and presents the link it read from its chain. The link is emitted by the OLD enclave during its prepare-upgrade flow, so it carries the old enclave's NSM attestation (PCRs = `from_pcrs`) and the old control key's 64-byte raw r||s ECDSA P-256 signature over the `UpgradePayload`. The synchronizer derives `old_key`/`new_key` from the payload (`from_pcrs`/`to_pcrs` hashes), requires `new_key` to equal the submitting session, and verifies the signature against the control pubkey frozen for `old_key` at its first registration (shipped). Control keys are P-256 throughout (no Ed25519).
- Multi-node mesh, PCR-bound mutual peer authentication (shipped, `mesh` module). Each node generates a per-boot P-256 mesh identity, dials its peers through `mesh-host` (vsock port 5009, `Open` frame then a single end-to-end ack byte), and listens for relayed-in peers on port 5008. Over each connection both ends run `Noise_NN`, then exchange and verify an NSM attestation document (nonce-bound to the handshake hash) plus a P-256 signature over that handshake hash by the announced mesh identity key (channel binding: a relay cannot splice a captured document onto a channel it terminates). The self-PCR allowlist admits a peer only if its attested PCR digest equals the node's own image measurements (un-upgradable v1, no successor-PCR admission). On top sits an id-correlated request/response layer (`Mesh::call(peer, payload)` plus an inbound `RequestHandler` hook) with a persistent per-peer channel that auto-reconnects with exponential backoff. Names are routing labels only; identity comes from attestation.
- Raft replication of the freshness map (shipped, `raft` module, behind the `raft` cargo feature; #119). [openraft](https://docs.rs/openraft) 0.9 drives an in-memory log + state machine (no persistence, per #16); the AppendEntries / Vote / InstallSnapshot RPCs ride the mesh's `MeshPayload` envelope via `Mesh::call`, decoded back into the local Raft by a `RaftRequestHandler` installed as the mesh's inbound handler. The leader does all cryptographic verification (session attestation at Register, the upgrade-link check at Transition) and replicates only the verified conclusions as a self-contained `ReplicatedOp { Register, Pin, Transition }`; followers re-apply those facts into the pure state machine deterministically, trusting the leader because mesh membership is already PCR-gated to bit-for-bit-identical peers (non-Byzantine). A snapshot serializes the entire pure-core state (committed projection plus observation sets), so a restarted node hydrates from the survivors over the same channel and behaves identically to one that replayed the whole log. `RaftHandle` exposes `client_write`, `is_leader` / `leader_name` (write-routing hints), and a linearizable read (`linearizable_get`, which refuses to serve stale data off a non-leader).
- Client RPC routing + node hydration wiring (shipped, `raft::serve` / `raft::forward` modules, behind the `raft` cargo feature; #120 / #121, slice 4). The customer listener now dispatches each attested-session request through a `SessionDispatch`: the single-node `Node` (no Raft) or the `ReplicatedDispatch` (Raft). On the leader, `Get` is served by `linearizable_get` (a freshness oracle never serves a follower-local read), `Pin` decides Register vs Pin from leader state and submits the matching `ReplicatedOp` (a concurrent-first-pin loser that is rejected `AlreadyRegistered` retries once as a Pin), and `Transition` runs the full #47 upgrade-link verification against the replicated state before submitting. A client may dial ANY node: a non-leader transparently FORWARDS the request to the current leader over the mesh (a new `MeshMessage::ForwardClient` envelope multiplexed into the same RPC payload namespace as the Raft RPCs) carrying the verified session facts (`PcrKey` + control pubkey + request), and relays the response, with a bounded retry across an in-progress election. No redirect variant is added to the customer wire protocol. The launcher wiring (`main.rs`) builds the mesh with a deferred `RaftRequestHandler`, stands up the `RaftHandle`, enables serving forwarded requests, and initializes the static membership on exactly the bootstrap node (id 0); already-initialized is benign, so a restart is safe. A node restarted with empty state rejoins as a follower and hydrates from the survivors (log replay or InstallSnapshot) via openraft's `loosen-follower-log-revert` mode. Cold-start precondition: durability is purely N-replica in-memory, so do not lose all three nodes simultaneously (cold-start recovery is #122, out of scope).

vsock-only.
